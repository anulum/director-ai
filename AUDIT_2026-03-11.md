# Director-AI Comprehensive Audit — 2026-03-11

## Executive Summary

Director-AI v3.8.0 is technically strong (75.8% AggreFact balanced accuracy, 14.6ms latency, streaming halt, 7 vector backends) but has **zero external users**. This audit identifies every weakness across 6 dimensions and proposes a prioritized action plan to reach top-tier production readiness.

**Overall readiness: 65%** — solid core, weak periphery.

## 5. HF DEMO WEAKNESSES

The current demo is **non-compelling** — it doesn't demonstrate the actual value proposition.

### Current Problems

1. **Uses `use_nli=False`** — the demo runs in heuristic mode only (~55% accuracy). Visitors see word-overlap scoring, not the actual NLI engine that delivers 75.8% accuracy. This is like demoing a Tesla with the engine off.

2. **No live LLM integration** — user must paste an LLM response manually. No "type a question, see the guardrail in action" flow. Every competitor demo (Guardrails AI, NeMo Guardrails) has a live chat interface.

3. **Fake streaming** — the streaming tab uses hardcoded token/score arrays, not actual streaming. Visitors cannot see real-time halt on their own input.

4. **No RAG demo** — the knowledge base is a 2-line textbox. No document upload, no PDF ingestion, no "bring your own data" flow.

5. **No customer-relevant examples** — "sky is blue" and "capital of France" are toy examples. No medical, finance, legal, customer support scenarios that show enterprise value.

6. **Static comparison tab** — only 3 hardcoded scenarios. Not interactive.

7. **Version not pinned** — `director-ai>=3.4.0,<4.0.0` — demo may break on upgrade.

8. **No error handling** — Gradio app crashes on invalid input with raw traceback.

### What the Demo Should Be

A **live chat interface** where:
1. User selects a domain (customer support, medical, legal, finance)
2. Pre-loaded knowledge base for that domain (FAQ, policy docs, medical guidelines)
3. User types a question → LLM generates answer → Director-AI scores it in real-time
4. Token-by-token visualization showing coherence degradation
5. Side-by-side: raw LLM output vs. Director-AI guarded output
6. Score breakdown with evidence chunks highlighted
7. Document upload for custom RAG

This requires a free LLM backend (HF Inference API, Ollama, or cached responses).

## 6. RAG & CUSTOMER DATA CAPABILITIES

### What Already Exists

Director-AI has a **complete RAG stack** that most users don't know about:

| Component | Status | Notes |
|-----------|--------|-------|
| In-memory keyword store | Working | `GroundTruthStore` — word overlap matching |
| In-memory vector store | Working | `InMemoryBackend` — TF-IDF-like word overlap |
| SentenceTransformer embeddings | Working | `SentenceTransformerBackend` — bge-large-en-v1.5 |
| ChromaDB | Working | `ChromaBackend` — local persistent vector DB |
| Pinecone | Working | `PineconeBackend` — cloud vector DB |
| Weaviate | Working | `WeaviateBackend` — cloud/self-hosted |
| Qdrant | Working | `QdrantBackend` — cloud/self-hosted |
| FAISS | Working | `FAISSBackend` — in-process dense search |
| Elasticsearch | Working | `ElasticsearchBackend` — hybrid BM25 + kNN |
| Cross-encoder reranking | Working | `RerankedBackend` — ms-marco-MiniLM-L-6-v2 |
| Bulk ingestion | Working | `VectorGroundTruthStore.ingest(texts)` |
| Multi-tenant isolation | Working | tenant_id filtering on all backends |
| Document chunking | **Missing** | No PDF/DOCX/HTML → chunks pipeline |
| Web scraping ingestion | **Missing** | No URL → facts pipeline |

### What's Missing for Customer RAG

1. **Document ingestion pipeline** — PDF, DOCX, HTML, CSV, JSON → chunk → embed → store. Currently users must pre-process documents themselves and call `.add()` per chunk.

2. **Automatic chunking** — no sentence/paragraph/semantic chunking. Users must split documents manually.

3. **CLI ingestion command** — `director-ai ingest --dir ./docs/ --backend chroma` doesn't exist. The `ingest` subcommand in cli.py is minimal.

4. **Metadata extraction** — no automatic extraction of source file, page number, section header from documents.

5. **Hybrid search default** — Elasticsearch backend has hybrid BM25+kNN, but it's not the default or easy to configure.

## 7. DOMAIN TRAINING OPPORTUNITIES

### Tier 1 — Highest Traction (train immediately)

| Domain | Why | Training Data | Market Signal |
|--------|-----|---------------|---------------|
| **Customer Support / FAQ** | Every SaaS company needs this. Lowest barrier to adoption. "Does the chatbot answer match our FAQ?" | Company FAQs, support tickets, policy docs. Use SNLI + domain-specific NLI pairs. | Zendesk AI, Intercom Fin, Ada — all ship without hallucination guarantees |
| **Medical / Clinical** | Highest stakes = highest willingness to pay. FDA pathway exists. | MedNLI (14K pairs), PubMedQA, HealthVer, MEDIQA-NLI | Hippocratic AI raised $53M. No open-source medical guardrail exists |
| **Legal / Compliance** | Contracts, regulations, HR policies. Lawyers demand accuracy. | ContractNLI (607 contracts), CUAD, LegalBench, RAGBench-Legal | Harvey AI valued at $1.5B. Legal hallucination = malpractice liability |

### Tier 2 — Strong Demand (train next)

| Domain | Why | Training Data |
|--------|-----|---------------|
| **Finance / Banking** | Regulatory compliance (SEC, FINRA). Wrong financial advice = liability. | FinanceBench, Financial PhraseBank, SEC filings NLI pairs |
| **E-commerce / Product** | Product descriptions, specs, pricing. "Does the AI match the product database?" | Amazon product QA, product attribute NLI |
| **HR / Internal Knowledge** | Company policies, benefits, onboarding. Every enterprise has this. | HR policy documents, benefits guides |

### Tier 3 — Specialized (train on demand)

| Domain | Why |
|--------|-----|
| **Education** | Exam prep, tutoring — correct answers matter |
| **Insurance** | Claims processing, policy explanation |
| **Government / Public Sector** | Citizen-facing chatbots, regulatory compliance |
| **Real Estate** | Property details, legal disclosures |

### Training Strategy

The SNLI training already running on JarvisLabs gives a general NLI foundation. Domain-specific fine-tuning should:

1. **Start with customer support** — easiest to get adoption data, fastest feedback loop, smallest domain vocabulary
2. **Add medical** — highest willingness to pay, existing benchmark suite, clear regulatory moat
3. **Add legal** — ContractNLI is ready-made, law firms have budget

Each domain needs:
- Domain-specific NLI pairs (2K-10K)
- Domain vocabulary augmentation
- Threshold tuning (medical: 0.75, legal: 0.68, support: 0.55)
- Evaluation benchmark per domain
- Pre-built knowledge base template
